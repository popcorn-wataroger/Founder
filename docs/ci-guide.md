# CI（自動チェック）ガイド

このドキュメントでは、CI（継続的インテグレーション）がどういう仕組みで動いていて、何を確認しているかを説明します。

---

## CIとは

PRを出すと、GitHubが自動でコードのチェックを実行してくれる仕組みです。

```
PRを出す
    ↓
GitHubがクラウド上に仮想マシン（ubuntu）を3台起動する
    ↓
3つのチェックを並列で実行する
    ↓
全部 ✓ → マージできる状態
いずれか ✗ → PR画面に赤く表示される（先に直す）
```

仮想マシンは毎回まっさらな状態から起動するため、「自分のPCでは動いた」が通用しない環境で確認できます。これがCIの大きなメリットです。

このプロジェクトには2つのワークフローがあります。

| ファイル | 役割 | 実行タイミング |
|---------|------|-------------|
| `.github/workflows/ci.yml` | コード品質チェック（lint・型・テスト） | PR・mainへのpush |
| `.github/workflows/codeql.yml` | セキュリティ脆弱性スキャン | PR・mainへのpush・毎週月曜 |

---

## 3つのコード品質チェック（ci.yml）

### 1. Lint & Format ― コードの書き方チェック

**何をしているか**

`ruff` というツールがコード全体を読んで、ルール違反とフォーマットのずれを検出します。

```bash
uv run ruff check .         # ルール違反を探す
uv run ruff format --check  # フォーマットのずれを探す
```

**検出できるもの**

| チェック内容 | 悪い例 |
|------------|--------|
| 未使用の import | `import os` と書いて一度も使っていない |
| 未定義の変数 | タイポで存在しない変数を参照している |
| import の順番 | 標準ライブラリ → サードパーティ → 自作の順になっていない |
| スペースのずれ | インデントがタブとスペースで混在している |

**エラーが出たら**

フォーマットのずれは自動修正できます。

```bash
uv run ruff check --fix .   # ルール違反を自動修正
uv run ruff format .        # フォーマットを自動修正
```

---

### 2. Type Check ― 型ヒントのチェック

**何をしているか**

`mypy` というツールがPythonの型ヒントを読んで、型の食い違いを検出します。

```bash
uv run mypy app/
```

**検出できるもの**

```python
# こういうミスを事前に発見できる例
def get_user_name(user_id: int) -> str:
    return 123  # str を返すと言ったのに int を返している → エラー
```

このプロジェクトは CLAUDE.md で「型ヒント必須」としているため、CIで自動的に担保しています。

**エラーが出たら**

エラーメッセージに「どのファイルの何行目か」が表示されるので、その箇所の型ヒントを確認してください。

---

### 3. Test & Smoke Test ― 動作確認

2段階に分かれています。

#### ① pytest（テスト）

```bash
uv run pytest -q --tb=short
```

テストコードを実行して、コードが期待通りに動くか確認します。現在 `tests/test_main.py` に1件あります。

```python
def test_root_returns_html():
    response = client.get("/")
    assert response.status_code == 200           # 200 が返るか
    assert "text/html" in response.headers[...]  # HTML が返るか
```

実際にサーバーを起動せず、FastAPI のテスト用クライアントで `/` を叩いています。

新しい API を追加したら、対応するテストを `tests/` に追加してください。

#### ② スモークテスト（起動確認）

```bash
uv run uvicorn app.main:app --port 8000 &  # サーバーをバックグラウンドで起動
curl -sf http://localhost:8000/            # HTTP リクエストを送って応答を確認
```

「電源を入れて煙が出ないか確認する」という意味で **スモークテスト** と呼ばれます。サーバーが起動できて最低限の応答が返るかだけを確認します。

pytest との違いはこうです。

| | pytest | スモークテスト |
|--|--------|-------------|
| サーバーを起動するか | しない | する |
| 確認していること | ロジックの正しさ | 「サーバーが動く」という事実 |
| 発見できるもの | コードのバグ | 起動クラッシュ・依存パッケージの欠落 |

## セキュリティスキャン（codeql.yml）

**何をしているか**

GitHubが開発した **CodeQL** というツールが、コードを静的解析してセキュリティ上の問題を検出します。

```
コードをスキャン
    ↓
脆弱性のパターンと照合
    ↓
問題があれば Security タブにアラートとして表示
```

対象言語はPython（バックエンド）とJavaScript（フロントエンド）の両方です。

**検出できる主な問題**

| 種類 | 例 |
|------|---|
| SQLインジェクション | ユーザー入力をそのままSQLに組み込んでいる |
| コマンドインジェクション | `subprocess` にユーザー入力を渡している |
| ハードコードされたシークレット | APIキーをコードに直書きしている |
| XSS（クロスサイトスクリプティング） | ユーザー入力をそのままHTMLに出力している |

**結果の見方**

CIと違い、アラートが出ても **PRは自動ではブロックされません**。結果はリポジトリの **Security タブ → Code scanning alerts** に一覧で表示されます。アラートが出たら内容を確認し、対応が必要かどうか判断してください。

毎週月曜にも定期スキャンが走るため、新しい脆弱性パターンが追加されたときに過去のコードが検出されることもあります。

---

## PR画面での見方

PR を出すと、画面下部にチェックの結果が表示されます。

```
✓ Lint & Format    — 問題なし
✓ Type Check       — 問題なし
✗ Test & Smoke Test — 失敗
```

`Details` リンクをクリックするとログが見えて、どのステップで何が起きたかを確認できます。

**CIが赤いままマージしてはいけません。** 直してから再 push するとCIが再実行されます。

## ローカルで事前に確認する

PR を出す前にローカルで確認しておくと、CIが赤くなるのを防げます。

**Claude Code の `/ci` コマンドを使う（推奨）**

Claude Code（このAIアシスタント）上で `/ci` と入力すると、4つのチェックを順番に自動で実行して結果を日本語で教えてくれます。エラーが出た場合は、その場で原因と修正方法も説明します。

**手動で確認する場合**

```bash
uv run ruff check .          # lint
uv run ruff format --check . # フォーマット確認
uv run mypy app/             # 型チェック
uv run pytest -q             # テスト
```

## よくある質問

**Q. CIが通っていれば動作確認しなくていい？**

いいえ。CIはコードの機械的なチェックであって、「画面が正しく表示される」「操作の流れが正しい」といった確認はできません。動作確認は引き続き自分で行ってください。

**Q. APIキーがないと動かないのでは？**

CI上では `.env` が存在しないため、ワークフローにダミー値を設定しています。テスト中に実際のAPIを叩くことはないので問題ありません。本物のAPIキーが必要なテストを書く場合は、GitHubの「Secrets」機能を使います（キーを直接ファイルに書いてはいけません）。

**Q. テストファイルはどこに書く？**

`tests/` ディレクトリの中に `test_` で始まるファイル名で作ります。pytest がそのルールでファイルを自動検出します。
