# 社内AIチャットボット — MVP要件定義書

## 1. プロダクト概要

NotebookLM的な社内ナレッジチャットボット。社長がアップロードしたソース（マニュアル、規定、社員個別資料など）をもとに、AIが社員の質問に回答する。

### コンセプト
- **社員向け：** LINE風チャットUIで社内ナレッジに即座にアクセス
- **社長向け：** ソース管理、チャットログ閲覧、スタッフ管理を一元化

---

## 2. ユーザーロールと認証

### ロール定義

| ロール | 対象 | アクセス範囲 |
|--------|------|-------------|
| 社員 | 全スタッフ | 社員用チャット画面のみ |
| 社長（管理者） | 1名 | 管理者画面（全機能） |

### 認証方式（MVP）
- CSVファイルで社員マスタ管理
- ログイン画面で社員コード＋パスワード（ダミーデータでOK）
- ロール判定：ログイン時にCSVのロールフィールドで振り分け

### 社員マスタCSV 項目例

```
社員コード, 名前, 部署, 性別, 生年月日, 家族構成, 入社日, 雇用形態, パスワード, ロール
```

## 3. 画面構成

### 3-1. 社員用画面：LINE風チャットUI

#### 機能
- テキスト入力 → AIが全社共通ソースをもとに回答
- チャット履歴はセッションごとに保持
- レスポンシブ対応（スマホでも使える想定）

#### AI回答の参照範囲
- 全社共通ソース：○（参照可）
- 自分の個別ソース：△（将来対応。DBには所有者フィールドを最初から持たせる）
- 他人の個別ソース：✕（絶対にアクセス不可）

#### UI要件
- メッセージの吹き出し表示（自分＝右、AI＝左）
- 送信ボタン＋Enterキー送信
- ローディングインジケーター（AI応答待ち）
- チャット履歴のスクロール

---

### 3-2. 管理者画面（PC用）

3つのメインセクションで構成。

#### A. ソース管理

| 項目 | 内容 |
|------|------|
| アップロード対応形式 | PDF, Word(.docx), PowerPoint(.pptx), テキスト(.txt), ウェブURL |
| ソース種別 | 「全社共通」または「社員個別」を選択してアップロード |
| 社員個別の場合 | 対象社員を紐付け |
| 操作 | アップロード、削除、一覧表示 |
| 表示情報 | ファイル名、アップロード日時、種別、対象社員（個別の場合） |

#### B. チャットログ

| 項目 | 内容 |
|------|------|
| 一覧表示 | 日付、時刻、社員名、トーク内容（抜粋） |
| フィルター | 社員名、日付範囲 |
| 詳細表示 | クリックで該当チャット全文を閲覧可能 |

#### C. スタッフ一覧 → 社員データ画面

**スタッフ一覧：**
- カード形式で社員を一覧表示（アイコン＋名前）
- クリックで社員データ画面に遷移

**社員データ画面（詳細）：**

| セクション | 内容 | MVP対応 |
|-----------|------|---------|
| 基本情報 | 名前、社員コード、部署、性別、生年月日、家族構成、入社日、雇用形態 | ○ |
| 最終ログイン履歴 | 最終ログイン日時 | ○ |
| 「このスタッフについてチャット」ボタン | 押下でAIチャット開始（後述） | ○ |
| 最近のトーク内容 | 直近N件のチャットログをプレビュー表示 | ○ |
| トーク全文閲覧 | クリックで全チャット履歴を閲覧（ページスクロール可能） | ○ |
| 給与グラフ | 給与推移のグラフ表示 | ✕（将来対応） |
| 過去のソース | その社員に紐付いたソース一覧＋ダウンロード機能 | ○ |
| ソース追加 | その社員への個別ソースアップロード | ○ |

#### 「このスタッフについてチャット」機能

2つの用途を兼ねる：

1. **AIチャット（メイン）：** 社長がその社員について質問 → AIが共通ソース＋その社員の個別ソースを参照して回答
   - 例：「奥村さんの直近の評価は？」→ 個別ソース（評価資料）から回答
2. **ログ閲覧：** その社員とボットの過去のチャット履歴も参照可能

## 4. データモデル（概念設計）

### テーブル構成

```
users（社員マスタ）
├── user_id（PK）
├── employee_code（社員コード）
├── name
├── department（部署）
├── gender（性別）
├── birth_date（生年月日）
├── family（家族構成）
├── hire_date（入社日）
├── employment_type（雇用形態）
├── role（social / admin）
├── password_hash
└── last_login_at

sources（ソース）
├── source_id（PK）
├── file_name
├── file_type（pdf / docx / pptx / txt / url）
├── file_path（ストレージパス or URL）
├── scope（common / individual）
├── owner_user_id（FK → users, NULLなら全社共通）
├── uploaded_at
└── uploaded_by（FK → users）

chat_sessions（チャットセッション）
├── session_id（PK）
├── user_id（FK → users）
├── started_at
└── context_type（general / staff_inquiry）
     ※ staff_inquiry = 社長が「このスタッフについてチャット」で開始

chat_messages（チャットメッセージ）
├── message_id（PK）
├── session_id（FK → chat_sessions）
├── role（user / assistant）
├── content
├── created_at
└── referenced_sources[]（参照したソースIDリスト）
```

### 権限マトリクス

| 操作 | 社員 | 社長 |
|------|------|------|
| 共通ソースを参照してAIに質問 | ○ | ○ |
| 自分の個別ソースを参照してAIに質問 | △（将来） | ○ |
| 他人の個別ソースを参照してAIに質問 | ✕ | ○ |
| ソースのアップロード・削除 | ✕ | ○ |
| チャットログの閲覧 | 自分のみ | 全員分 |
| 社員データの閲覧 | ✕ | ○ |

## 5. 非機能要件

### セキュリティ
- チャットボット利用規約に「業務用チャットのログは管理目的で記録されます」と明記
- 初回ログイン時に同意チェック
- 個別ソースの権限制御（社員が他人のデータにアクセスできないこと）

### 想定規模
- 社員数：100人以下
- 同時接続：数十人程度

### 技術スタック

| 領域 | 採用技術 | 備考 |
|------|---------|------|
| フロントエンド | HTML / CSS / JavaScript | フレームワークなし、バニラ構成 |
| バックエンド | Python（FastAPI） | 非同期対応、OpenAI SDK との相性◎ |
| AI基盤 | OpenAI API | GPT-4o 等 |
| ベクトルDB | Qdrant | 軽量・セットアップ容易・OSSで無料利用可 |
| ファイルストレージ | Google Cloud Storage (GCS) | ソースファイルの保管 |
| ホスティング | Google Cloud Platform (GCP) | Cloud Run + Cloud SQL 等 |
| チャット履歴保持期間 | 無期限 | 削除は手動対応 |
| リポジトリ | `Founder` | GitHub |

※ ベクトルDBは Qdrant を推奨。理由：Docker一発で起動可能、REST APIがシンプル、Python SDKが充実、GCP上でも Cloud Run でホスト可能。pgvector（PostgreSQL拡張）も候補だが、Cloud SQL と同居させる場合はセットアップがやや複雑。

### 技術的前提
- Webアプリケーション（LINE風、ネイティブアプリではない）
- 外部システム連携：MVPでは不要
- 社員マスタ：CSVインポート（ダミーデータ）

## 6. フェーズ計画

### Phase 1（MVP）— 本ドキュメントのスコープ
- 社員用LINE風チャットUI
- 管理者画面（ソース管理、ログ閲覧、スタッフ一覧・詳細）
- AIチャットエンジン（権限ベースのソース参照）
- CSV認証
- 「このスタッフについてチャット」

### Phase 2（拡張）
- 給与グラフ表示
- 本人への個別データ公開（自分の評価・給与を自分で閲覧）
- 検索・フィルター強化

### Phase 3（発展）
- Google Workspace / Slack連携
- 通知機能
- 分析ダッシュボード（よくある質問の傾向など）
